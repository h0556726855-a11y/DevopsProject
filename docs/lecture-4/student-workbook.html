Core Concepts

Concept 1

Infrastructure as Code (IaC)

Infrastructure is defined in version-controlled text files rather than clicked together in a console. Every cloud resource — VPC, subnets, EKS cluster — is reproducible from a single terraform apply, and equally teardownable with terraform destroy. The state file (terraform.tfstate) is Terraform's memory of what it built.

Concept 2

Terraform Workflow

Three commands form the core loop: init downloads provider plugins, plan shows a dry-run diff of what will change, and apply executes the changes against AWS. Never skip the plan step — it is your last chance to catch accidental deletions before real resources are affected.

Concept 3

Amazon EKS

Elastic Kubernetes Service runs the Kubernetes control plane (API server, etcd, scheduler) as a managed AWS service. You pay $0.10/hr for the control plane regardless of node count. Worker nodes are regular EC2 instances in your account — you own their costs and security patches. EKS integrates with IAM for authentication and ALB/NLB for load balancing.

Concept 4

LoadBalancer Service Type

When a Kubernetes Service's type: LoadBalancer is applied to an EKS cluster, the cloud controller manager automatically provisions an AWS Network Load Balancer (NLB) and wires it to the service's endpoints. The NLB gets a public DNS name within 2–3 minutes. This replaces the NodePort approach from local development.

Concept 5

Deploy Stage in CI/CD

The pipeline now has three stages: build (Docker image), verify (tests, lint), deploy (kubectl apply to EKS). The deploy stage needs AWS credentials injected as CI/CD variables, not stored in code. It runs aws eks update-kubeconfig to authenticate kubectl, then applies the Kubernetes manifests.

  2. 2

    Write versions.tf — pin provider versions

    Pinning providers prevents unexpected upgrades from breaking your infrastructure.

    # terraform/versions.tf
terraform {
  required_version = ">= 1.6"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
    kubernetes = {
      source  = "hashicorp/kubernetes"
      version = "~> 2.0"
    }
  }
}
  3. 3

    Write variables.tf — parameterise the cluster

    Input variables allow the same Terraform code to target different regions or instance sizes without editing main.tf.

    # terraform/variables.tf
variable "aws_region" {
  description = "AWS region to deploy into"
  type        = string
  default     = "us-east-1"
}

variable "cluster_name" {
  description = "Name for the EKS cluster"
  type        = string
  default     = "shoplist-eks"
}

variable "node_instance_type" {
  description = "EC2 instance type for worker nodes"
  type        = string
  default     = "t3.micro"
}

variable "desired_node_count" {
  description = "Number of worker nodes"
  type        = number
  default     = 2
}
  4. 4

    Write main.tf — VPC, subnets, EKS cluster

    This is the core infrastructure definition. It creates a VPC with two public subnets and an EKS cluster with a managed node group.

    # terraform/main.tf
provider "aws" {
  region = var.aws_region
}

# ── Networking ────────────────────────────────
resource "aws_vpc" "shoplist" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true
  enable_dns_support   = true

  tags = { Name = "${var.cluster_name}-vpc" }
}

resource "aws_internet_gateway" "igw" {
  vpc_id = aws_vpc.shoplist.id
  tags   = { Name = "${var.cluster_name}-igw" }
}

resource "aws_subnet" "public" {
  count             = 2
  vpc_id            = aws_vpc.shoplist.id
  cidr_block        = "10.0.${count.index}.0/24"
  availability_zone = data.aws_availability_zones.available.names[count.index]
  map_public_ip_on_launch = true

  tags = {
    Name                                        = "${var.cluster_name}-public-${count.index}"
    "kubernetes.io/cluster/${var.cluster_name}"  = "owned"
    "kubernetes.io/role/elb"                     = "1"
  }
}

resource "aws_route_table" "public" {
  vpc_id = aws_vpc.shoplist.id
  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.igw.id
  }
  tags = { Name = "${var.cluster_name}-rt" }
}

resource "aws_route_table_association" "public" {
  count          = 2
  subnet_id      = aws_subnet.public[count.index].id
  route_table_id = aws_route_table.public.id
}

data "aws_availability_zones" "available" {
  state = "available"
}

# ── IAM Roles ─────────────────────────────────
data "aws_iam_policy_document" "eks_assume" {
  statement {
    actions = ["sts:AssumeRole"]
    principals {
      type        = "Service"
      identifiers = ["eks.amazonaws.com"]
    }
  }
}

resource "aws_iam_role" "eks_cluster" {
  name               = "${var.cluster_name}-cluster-role"
  assume_role_policy = data.aws_iam_policy_document.eks_assume.json
}

resource "aws_iam_role_policy_attachment" "eks_cluster" {
  role       = aws_iam_role.eks_cluster.name
  policy_arn = "arn:aws:iam::aws:policy/AmazonEKSClusterPolicy"
}

data "aws_iam_policy_document" "node_assume" {
  statement {
    actions = ["sts:AssumeRole"]
    principals {
      type        = "Service"
      identifiers = ["ec2.amazonaws.com"]
    }
  }
}

resource "aws_iam_role" "node_group" {
  name               = "${var.cluster_name}-node-role"
  assume_role_policy = data.aws_iam_policy_document.node_assume.json
}

resource "aws_iam_role_policy_attachment" "node_worker" {
  role       = aws_iam_role.node_group.name
  policy_arn = "arn:aws:iam::aws:policy/AmazonEKSWorkerNodePolicy"
}

resource "aws_iam_role_policy_attachment" "node_cni" {
  role       = aws_iam_role.node_group.name
  policy_arn = "arn:aws:iam::aws:policy/AmazonEKS_CNI_Policy"
}

resource "aws_iam_role_policy_attachment" "node_ecr" {
  role       = aws_iam_role.node_group.name
  policy_arn = "arn:aws:iam::aws:policy/AmazonEC2ContainerRegistryReadOnly"
}

# ── EKS Cluster ───────────────────────────────
resource "aws_eks_cluster" "shoplist" {
  name     = var.cluster_name
  role_arn = aws_iam_role.eks_cluster.arn
  version  = "1.29"

  vpc_config {
    subnet_ids = aws_subnet.public[*].id
  }

  depends_on = [aws_iam_role_policy_attachment.eks_cluster]
}

resource "aws_eks_node_group" "workers" {
  cluster_name    = aws_eks_cluster.shoplist.name
  node_group_name = "workers"
  node_role_arn   = aws_iam_role.node_group.arn
  subnet_ids      = aws_subnet.public[*].id

  instance_types = [var.node_instance_type]

  scaling_config {
    desired_size = var.desired_node_count
    max_size     = 3
    min_size     = 1
  }

  depends_on = [
    aws_iam_role_policy_attachment.node_worker,
    aws_iam_role_policy_attachment.node_cni,
    aws_iam_role_policy_attachment.node_ecr,
  ]
}

    Add the deploy stage to .gitlab-ci.yml

    The deploy stage authenticates kubectl using CI/CD variables and applies the manifests. Add AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, and AWS_DEFAULT_REGION as protected CI/CD variables in GitLab Settings → CI/CD → Variables.

    # .gitlab-ci.yml — append the deploy stage
stages:
  - build
  - verify
  - deploy

deploy-to-eks:
  stage: deploy
  image: amazon/aws-cli:latest
  before_script:
    - curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"
    - chmod +x kubectl && mv kubectl /usr/local/bin/
    - aws eks update-kubeconfig
        --region "$AWS_DEFAULT_REGION"
        --name "shoplist-eks"
  script:
    - kubectl apply -f kubernetes/
    - kubectl rollout status deployment/backend --timeout=120s
    - kubectl rollout status deployment/frontend --timeout=120s
  environment:
    name: production
  only:
    - main

Interview Questions & Answers

What is the difference between Terraform state and the actual infrastructure?
The state file (terraform.tfstate) is a local JSON record of what Terraform thinks it built. The actual infrastructure is whatever AWS currently has. These can drift if someone manually changes a resource in the console without going through Terraform. Running terraform refresh (or terraform plan which does this implicitly) reconciles state with reality and shows any drift. This is why direct console changes are discouraged — they create hidden divergence between code and reality.
Why does EKS cost $0.10/hour even with no workloads running?
The EKS control plane (API server, etcd, scheduler, controller manager) runs on AWS-managed infrastructure that AWS dedicates to your cluster. That compute is billable regardless of whether you have any pods scheduled. Worker nodes (EC2) are separate — you only pay for those while they are running. This is why terraform destroy is essential: it deletes the cluster (stopping the $0.10/hr charge) and terminates the EC2 worker nodes.
What happens to the LoadBalancer Service when you run kubectl delete service frontend?
The Kubernetes cloud controller manager watches for Service deletion events and calls the AWS API to delete the underlying Network Load Balancer. This is an important side effect to understand: deleting a Kubernetes resource can trigger deletion of a real AWS resource. The reverse is also true — if someone manually deletes the NLB in the AWS console, the Service will enter a broken state. All AWS resource lifecycle changes should flow through Kubernetes or Terraform, not the console.
What is the purpose of the subnet tags kubernetes.io/role/elb: "1"?
When Kubernetes provisions a LoadBalancer Service, the AWS cloud controller manager needs to know which subnets to place the load balancer in. It discovers eligible subnets by looking for specific tags. The tag kubernetes.io/role/elb: "1" marks a subnet as available for internet-facing (public) load balancers. Without this tag, EKS cannot create the NLB and the Service will remain in a pending state indefinitely — one of the most common gotchas when setting up EKS from scratch.
How do you safely pass AWS credentials to a GitLab CI/CD pipeline without hardcoding them?
GitLab CI/CD Variables (Settings → CI/CD → Variables) store secrets server-side and inject them as environment variables at runtime. Mark them as Protected (only available on protected branches) and Masked (never printed in job logs). The job uses them as $AWS_ACCESS_KEY_ID and $AWS_SECRET_ACCESS_KEY — the AWS CLI and SDKs automatically read these standard env var names. An even better production approach is OIDC federation: GitLab generates a short-lived JWT that AWS IAM trusts, eliminating long-lived keys entirely.

Destroy Protocol — Run This Before You Finish

Run every command in sequence. Verify each one succeeds before continuing. The entire sequence takes about 5 minutes.

1. Verify what is still running

# Confirm kubectl is pointed at the right cluster
kubectl config current-context

# List any LoadBalancer services (NLBs cost money)
kubectl get svc --all-namespaces | grep LoadBalancer

2. Delete Kubernetes workloads first

# Deleting services before terraform destroy
# ensures AWS cleans up NLBs gracefully
kubectl delete -f kubernetes/ --ignore-not-found=true

# Wait for LoadBalancer deletion (~60 seconds)
kubectl get svc --watch

3. Run terraform destroy

cd terraform/

# Preview what will be destroyed — confirm 14 resources
terraform plan -destroy

# Destroy all infrastructure
terraform destroy -auto-approve

4. Confirm destruction in AWS Console

After terraform destroy completes, verify manually:

  • AWS Console → EKS → Clusters: list should be empty
  • AWS Console → EC2 → Instances: no instances in "running" state from this lab
  • AWS Console → EC2 → Load Balancers: no NLBs from this lab
  • AWS Console → VPC → Your VPCs: shoplist-eks-vpc should be gone
  • AWS Billing → Bills: verify no unexpected charges within 15 minutes

5. Emergency manual destroy (if Terraform fails)

# If terraform destroy errors, delete the cluster directly
aws eks delete-nodegroup \
  --cluster-name shoplist-eks \
  --nodegroup-name workers \
  --region us-east-1

# Wait for node group deletion, then delete cluster
aws eks delete-cluster \
  --name shoplist-eks \
  --region us-east-1

# Verify cluster is gone
aws eks list-clusters --region us-east-1
