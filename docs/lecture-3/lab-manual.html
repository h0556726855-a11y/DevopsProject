1

Inspect the Pipeline

Easy Observation ~10 min

Open .gitlab-ci.yml at the project root and answer each question in writing or aloud.

Questions

Q1: How many jobs are defined? List them by name.
Expected: 4 — build-backend, build-frontend, test-backend, push-images.
Q2: Which stage does each job belong to?
build-backend → build. build-frontend → build. test-backend → test. push-images → push.
Q3: Two jobs run in parallel. Which two, and why?
build-backend and build-frontend. Both belong to the build stage. Jobs within the same stage run in parallel.
Q4: The push-images job has an only: block. What happens when you push to feature/lecture-3?
push-images is skipped. It only runs on dev or main. Build and test still run. This is expected behavior.
Q5: What happens if you remove the services: block from a job?
The first docker command fails with: Cannot connect to the Docker daemon at unix:///var/run/docker.sock. Is the docker daemon running?
Q6: Why does test-backend rebuild the image even though build-backend already built it?
Each job runs in a fresh isolated container. The image from build-backend is gone — different container, different dind daemon. The rebuild is required.
Q7: What is $CI_COMMIT_SHORT_SHA, and what value would it have for commit hash abc123456789?
First 8 characters of the git commit hash. Value: abc12345.
2

Remove dind and Observe Failure

Medium Break-and-fix ~15 min

Part A — Introduce the break

Edit .gitlab-ci.yml. In test-backend, remove the services: block:

Broken version (test-backend)
test-backend:
  stage: test
  image: docker:24
  script:
    - docker build -t $BACKEND_IMAGE:$IMAGE_TAG app/backend/
    - docker run --rm $BACKEND_IMAGE:$IMAGE_TAG python -c "from flask import Flask; import psycopg2; print('All dependencies OK')"
git add .gitlab-ci.yml
git commit -m "exercise: remove dind from test-backend"
git push origin feature/lecture-3

Part B — Observe and record the failure

Open GitLab → CI/CD → Pipelines. Watch the new pipeline. Answer:

Observe

1. Which stage fails?
2. What is the exact error message on the first failing line?
3. Does push-images run? Why or why not?

Expected observations

test-backend fails. Error: Cannot connect to the Docker daemon at unix:///var/run/docker.sock. Is the docker daemon running?
push-images does not run — test stage failed, push stage cancelled. This is the safety property.

Part C — Restore the fix

Restore the services: block to test-backend:

test-backend:
  stage: test
  image: docker:24
  services:
    - docker:24-dind
  script:
    - docker build -t $BACKEND_IMAGE:$IMAGE_TAG app/backend/
    - docker run --rm $BACKEND_IMAGE:$IMAGE_TAG python -c "from flask import Flask; import psycopg2; print('All dependencies OK')"
git add .gitlab-ci.yml
git commit -m "exercise: restore dind in test-backend"
git push origin feature/lecture-3
Expected

Pipeline passes all 3 stages. push-images skipped (feature branch). This is correct.

3

Change to :latest Tag and Observe Registry

Medium Remove-and-observe ~15 min

Part A — Switch to :latest

Edit the IMAGE_TAG variable:

Modified (using :latest)
variables:
  BACKEND_IMAGE: $CI_REGISTRY_IMAGE/backend
  FRONTEND_IMAGE: $CI_REGISTRY_IMAGE/frontend
  IMAGE_TAG: latest
git add .gitlab-ci.yml
git commit -m "exercise: switch to :latest tag"
git push origin dev

Wait for the full pipeline including push-images to complete.

Part B — Observe the registry

Open GitLab → Deploy → Container Registry → backend.

Observe

1. How many tags are listed?
2. From the tag name alone, can you tell which git commit built this image?
3. Make one more commit and push to dev. How many tags are listed now?

Expected observations

After first push: 1 tag named latest.
After second push: still 1 tag — the previous image was overwritten.
From the tag latest, you cannot determine which commit built the image. Rollback is impossible without extra metadata.

Part C — Restore SHA tagging

variables:
  BACKEND_IMAGE: $CI_REGISTRY_IMAGE/backend
  FRONTEND_IMAGE: $CI_REGISTRY_IMAGE/frontend
  IMAGE_TAG: $CI_COMMIT_SHORT_SHA
git add .gitlab-ci.yml
git commit -m "exercise: restore SHA tagging"
git push origin dev

Verify in the registry: the new tag matches git log --oneline -1 (8-char prefix).

★

Add a yamllint Validation Stage (Bonus)

Optional Extension ~20 min

Background

yamllint is a Python tool that validates YAML files for syntax errors and style issues. Adding it as a pre-build stage catches broken YAML before wasting runner time.

Task

Add a validate stage before build, with a lint-yaml job:

stages:
  - validate
  - build
  - test
  - push

lint-yaml:
  stage: validate
  image: python:3.11-slim
  script:
    - pip install yamllint --quiet
    - yamllint .gitlab-ci.yml

Note: no services: needed — this job does not use Docker.

Verify

Expected pipeline on feature branch

lint-yaml ✓ (validate stage) → build-backend ✓ + build-frontend ✓ → test-backend ✓ → push-images — (skipped)

To test the lint: introduce a deliberate indentation error, push, observe lint-yaml fail and build stage cancel. Restore the correct indentation to make lint pass again.
