Core Concepts

Concept 1 — CI/CD

What it is: Continuous Integration means every git push triggers an automated build and test. Continuous Delivery means passing builds are automatically packaged and pushed to a registry.

Why it exists: After Lecture 2, deploying a code change requires 4 manual steps. CI/CD replaces 3 of them with one git push. The remaining manual step (kubectl apply) is automated in Lecture 4.

How it works: GitLab reads .gitlab-ci.yml on every push, creates a pipeline, and dispatches jobs to runners. Each job runs in a fresh container, executes its script, and reports pass/fail.

Common mistake: Thinking CI/CD means automatic deployment. After this lecture, push is automated; kubectl apply is still manual.
Concept 2 — GitLab Runner

What it is: A process that picks up pipeline jobs and executes them in isolated containers.

Why it exists: GitLab orchestrates pipelines but doesn't execute jobs. Runners are the execution layer — they can run on any machine.

How it works: Runner picks up a queued job, starts a container from image:, clones the repo, runs script: steps, reports exit code. Container is destroyed when done.

Common mistake: Expecting the image from one job to be available in the next. Each job = fresh container. State does not persist between jobs.
Concept 3 — Docker-in-Docker (dind)

What it is: A Docker daemon running inside a Docker container, enabling CI jobs to run docker build inside a pipeline.

Why it exists: A GitLab job runs inside a container (image: docker:24), which has the Docker CLI but no daemon. services: - docker:24-dind adds the daemon as a sidecar.

How it works: The dind sidecar starts a daemon; GitLab sets DOCKER_HOST so the CLI connects to it. All docker commands in the script run against this daemon.

Common mistake: Forgetting services: - docker:24-dind. Result: Cannot connect to the Docker daemon at unix:///var/run/docker.sock.
Concept 4 — Container Registry

What it is: A server that stores and serves Docker images by name and tag. GitLab's built-in registry is at registry.gitlab.com/[group]/[project]/[image]:[tag].

Why it exists: Without a registry, images exist only on the machine that built them. The registry makes images accessible to any machine — including Kubernetes clusters — without local imports.

How it works: docker login → docker push uploads. Kubernetes pulls via the image: field in the deployment manifest.

Common mistake: Running docker push without docker login first. Result: denied: access forbidden.
Concept 5 — SHA Image Tagging

What it is: Using the git commit's 8-character SHA as the Docker image tag via $CI_COMMIT_SHORT_SHA, instead of :latest.

Why it exists: :latest is overwritten on every push. SHA tags are permanent and unique — each image tag points to exactly one commit.

How it works: GitLab sets $CI_COMMIT_SHORT_SHA to the first 8 hex chars of the commit hash. Setting IMAGE_TAG: $CI_COMMIT_SHORT_SHA makes it available to all jobs.

Common mistake: Using :latest. After 10 pushes, all 10 images share one tag — no traceability, no rollback.

What You Build — Pipeline Flow

git push (feature/lecture-3 or dev) ↓ .gitlab-ci.yml read by GitLab ↓ ┌─────────────────────────────────────────────┐ │ stage: build │ │ build-backend (docker build → backend) │ │ build-frontend (docker build → frontend) │ └─────────────────────────────────────────────┘ ↓ ┌─────────────────────────────────────────────┐ │ stage: test │ │ test-backend (dependency import check) │ └─────────────────────────────────────────────┘ ↓ (only on dev / main) ┌─────────────────────────────────────────────┐ │ stage: push │ │ push-images (docker login + push both) │ └─────────────────────────────────────────────┘ ↓ GitLab Container Registry registry.gitlab.com/.../backend:[sha] registry.gitlab.com/.../frontend:[sha] ↓ (manual kubectl apply) k3d cluster — pods use registry images

Step-by-Step Instructions

1

Enable Container Registry and verify runner

GitLab → Settings → General → Visibility → Container Registry → Enabled.
GitLab → Settings → CI/CD → Runners → at least one shared runner visible.

2

Create .gitlab-ci.yml with stages and variables

stages:
  - build
  - test
  - push

variables:
  BACKEND_IMAGE: $CI_REGISTRY_IMAGE/backend
  FRONTEND_IMAGE: $CI_REGISTRY_IMAGE/frontend
  IMAGE_TAG: $CI_COMMIT_SHORT_SHA
3

Add the build stage

build-backend:
  stage: build
  image: docker:24
  services:
    - docker:24-dind
  script:
    - docker build -t $BACKEND_IMAGE:$IMAGE_TAG app/backend/

build-frontend:
  stage: build
  image: docker:24
  services:
    - docker:24-dind
  script:
    - docker build -t $FRONTEND_IMAGE:$IMAGE_TAG app/frontend/
4

Add the test stage

test-backend:
  stage: test
  image: docker:24
  services:
    - docker:24-dind
  script:
    - docker build -t $BACKEND_IMAGE:$IMAGE_TAG app/backend/
    - docker run --rm $BACKEND_IMAGE:$IMAGE_TAG python -c "from flask import Flask; import psycopg2; print('All dependencies OK')"
Expected output
All dependencies OK
5

Add the push stage

push-images:
  stage: push
  image: docker:24
  services:
    - docker:24-dind
  script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
    - docker build -t $BACKEND_IMAGE:$IMAGE_TAG app/backend/
    - docker build -t $FRONTEND_IMAGE:$IMAGE_TAG app/frontend/
    - docker push $BACKEND_IMAGE:$IMAGE_TAG
    - docker push $FRONTEND_IMAGE:$IMAGE_TAG
  only:
    - dev
    - main
6

Push to feature branch and verify pipeline

git add .gitlab-ci.yml
git commit -m "ci: add GitLab CI/CD pipeline with build, test, push stages"
git push origin feature/lecture-3
Expected pipeline result
build-backend ✓ • build-frontend ✓ • test-backend ✓ • push-images — (skipped)
7

Merge to dev — trigger full pipeline with push

git checkout dev
git merge feature/lecture-3
git push origin dev
Expected pipeline result
All 4 jobs: build-backend ✓ build-frontend ✓ test-backend ✓ push-images ✓
8

Update Kubernetes manifests

In both kubernetes/backend-deployment.yml and kubernetes/frontend-deployment.yml, update the image and pull policy:

image: registry.gitlab.com/YOUR_GROUP/devops-tikshuv-project/backend:YOUR_SHA
imagePullPolicy: Always

Replace YOUR_GROUP and YOUR_SHA with values from the GitLab Container Registry and git log --oneline -1.

9

Apply and verify

kubectl apply -f kubernetes/backend-deployment.yml
kubectl apply -f kubernetes/frontend-deployment.yml
kubectl rollout status deployment/backend -n shoplist
kubectl rollout status deployment/frontend -n shoplist
kubectl describe pod -n shoplist -l app=backend | grep Image:
curl http://localhost:30080/api/health
Expected
Rollout successful. Image: shows registry.gitlab.com/... Health check: {"status": "ok"}

Interview Questions

Q: What is the difference between CI and CD?
CI — Continuous Integration — means every commit triggers automated build and test. CD — Continuous Delivery — means passing builds are automatically packaged and deployed. In this lecture we implement CI (build + test) and the push to registry (first half of CD). Automated kubectl apply is CD's second half — Lecture 4.
Q: What does $CI_COMMIT_SHORT_SHA contain, and why use it instead of :latest?
It contains the first 8 characters of the git commit hash. Using it as the image tag means every commit produces a permanent, uniquely identified image. With :latest, every push overwrites the same tag — you lose the ability to trace which code is running and cannot roll back to a specific version.
Q: What is Docker-in-Docker, and why is it required?
Docker-in-Docker is a Docker daemon running inside a container. A GitLab CI job runs inside docker:24 (has Docker CLI) but has no daemon by default. Adding services: - docker:24-dind starts a daemon as a sidecar. Without it, docker build fails with "Cannot connect to the Docker daemon."
Q: After pushing a new image to the registry, what commands update the running Kubernetes pod?
(1) Update image: in the deployment YAML with the new registry path and SHA. (2) kubectl apply -f kubernetes/[service]-deployment.yml. (3) kubectl rollout status deployment/[service] -n shoplist to confirm completion.
Q: What is the purpose of the test stage, and why must it precede push?
The test stage verifies the image is internally valid (pip install succeeded) before it enters the registry. It runs an import check inside the built image. It precedes push because the pipeline's guarantee is: only valid artifacts reach the registry. A broken image that never enters the registry cannot be accidentally deployed.

Appendix G — GitHub Actions (Alternative to GitLab CI)

Use this appendix if your project is hosted on GitHub instead of GitLab. The concepts are identical — see cicd-primer.html for the side-by-side comparison. Only the file location, YAML syntax, and registry URL differ.

The workflow file already exists in this repo

.github/workflows/build-and-push.yml — you do not need to write it from scratch. It is the GitHub equivalent of .gitlab-ci.yml.

G2 — Enable package write permissions

GitHub → your repo → Settings → Actions → General → Workflow permissions → select "Read and write permissions" → Save.

Without this: the push step fails with denied: installation not allowed to Write organization package.

G3 — Push and observe the pipeline

git add .github/workflows/build-and-push.yml
git commit -m "ci: add GitHub Actions workflow for build and push"
git push origin main

Open GitHub → your repo → Actions tab. Watch the three jobs run.

Expected result
build-backend → passed
build-frontend → passed
verify-registry → passed

G4 — Find images in ghcr.io

After the workflow passes: GitHub → your profile → Packages. You should see shoplist-backend and shoplist-frontend, each tagged with the 8-character commit SHA.

git log --oneline -1
# Example: a1b2c3d4 ci: add GitHub Actions workflow...
# Image URLs:
# ghcr.io/YOUR_GITHUB_USERNAME/shoplist-backend:a1b2c3d4
# ghcr.io/YOUR_GITHUB_USERNAME/shoplist-frontend:a1b2c3d4

G5 — Update Kubernetes manifests

Edit kubernetes/backend-deployment.yml and kubernetes/frontend-deployment.yml:

image: ghcr.io/YOUR_GITHUB_USERNAME/shoplist-backend:YOUR_SHA
imagePullPolicy: Always

Replace YOUR_GITHUB_USERNAME (lowercase) and YOUR_SHA with the 8-character SHA from G4.

G6 — Make packages public or create an imagePullSecret

ghcr.io packages are private by default. Kubernetes needs credentials to pull them.

Option A — Make public (easiest)

GitHub → Packages → select package → Package settings → Change visibility → Public. Repeat for both images. No Kubernetes credentials needed.

Option B — imagePullSecret

Create a GitHub personal access token (read:packages scope), then create a Kubernetes secret and add imagePullSecrets: to each deployment spec.

# Option B — create the pull secret
kubectl create secret docker-registry ghcr-secret \
  --docker-server=ghcr.io \
  --docker-username=YOUR_GITHUB_USERNAME \
  --docker-password=YOUR_PERSONAL_ACCESS_TOKEN \
  -n shoplist
