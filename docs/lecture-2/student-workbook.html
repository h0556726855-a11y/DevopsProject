Part 1 — Core Concepts

1

Pod and Deployment

WHAT
A Pod is the smallest deployable unit — one or more containers sharing a network interface. A Deployment declares a desired number of pod replicas and manages their lifecycle automatically.
WHY
Docker Compose has no mechanism for automatic container restart on failure. Kubernetes Deployments implement a control loop: actual state is continuously compared to desired state. If a pod crashes, the Deployment creates a replacement immediately.
HOW
A Deployment creates a ReplicaSet, which creates and monitors Pods. spec.replicas declares how many pods should run. selector.matchLabels links the Deployment to its pods. Missing pods are created automatically.
COMMON MISTAKE
Running bare pods instead of Deployments. Any pod created without a Deployment is unmanaged — if it crashes, it stays crashed. Always use Deployments.
2

Service

WHAT
A Service is a stable network endpoint that routes traffic to pods matching a label selector. Services provide fixed DNS names and IP addresses that remain constant even as pod IPs change.
WHY
Pods are ephemeral — when a pod restarts, it gets a new IP. Services solve this with a stable name. nginx.conf with proxy_pass http://backend:5000/ works unchanged because "backend" always resolves to the current backend pod via the Service.
HOW
TypeReachable FromUsed In ShopList
ClusterIP (default)Inside cluster onlypostgres, backend
NodePortHost machine via port 30000–32767frontend (:30080)
LoadBalancerExternal IP (cloud)Lecture 4 only
COMMON MISTAKE
Referring to pods by IP address. Pod IPs are ephemeral and change on every restart. Always use the Service name for inter-pod communication.
3

PersistentVolumeClaim

WHAT
A PVC is a request for persistent storage. Kubernetes binds it to a PersistentVolume. The PVC — and its data — survives pod deletions.
WHY
Container filesystems are ephemeral. When a pod is deleted, its filesystem is destroyed. PVCs separate the storage lifecycle from the pod lifecycle. The PVC exists independently of any pod until explicitly deleted.
HOW
Declare a PVC with a size (1Gi) and access mode (ReadWriteOnce). Mount it in the pod spec at /var/lib/postgresql/data. When the pod is deleted, the PVC remains. The replacement pod mounts the same PVC and finds all data intact.
COMMON MISTAKE
Confusing pod deletion with data loss. Deleting a pod in Kubernetes is safe — the Deployment creates a replacement that mounts the same PVC. To permanently delete data: kubectl delete pvc postgres-pvc -n shoplist. This is the equivalent of docker compose down -v.
4

Secret

WHAT
A Secret stores sensitive data — passwords, tokens, keys — in base64-encoded form, separately from other configuration, with independent RBAC access control.
WHY
Secrets can be restricted via Kubernetes RBAC — only specific service accounts need read access. ConfigMaps are readable by everyone in the namespace. The separation enables least-privilege access control.
HOW
echo -n "shoppass" | base64      # → c2hvcHBhc3M=
echo "c2hvcHBhc3M=" | base64 -d  # → shoppass
Secret values are base64-encoded strings, not encrypted. They can be mounted as environment variables or files.
COMMON MISTAKE
Thinking base64 is encryption — it is not. The security comes from RBAC restricting who can read the Secret object, not from the encoding. Never commit Secret YAML files to git.
5

ConfigMap

WHAT
A ConfigMap stores non-sensitive configuration — text files, key-value pairs — that can be mounted into pods as environment variables or as files on the filesystem.
WHY
Hardcoding configuration inside images means rebuilding for every configuration change. ConfigMaps allow images to be generic and reusable — configuration is provided at runtime.
HOW
In ShopList, postgres-init ConfigMap contains init.sql. It is mounted at /docker-entrypoint-initdb.d. PostgreSQL runs all .sql files in that directory on first startup.
COMMON MISTAKE
Putting secrets in ConfigMaps because it is simpler. ConfigMaps are readable by any pod and user in the namespace. Database passwords and API keys must always go in Secrets.
6

Namespace

WHAT
A Namespace is a logical partition within a Kubernetes cluster that provides isolated scopes for names, access control, and resource quotas. All ShopList resources live in shoplist.
WHY
Without namespaces, all resources in a cluster share one flat scope — naming collisions and impossible per-application access control. Namespaces enable multi-tenant clusters.
HOW
Every kubectl command requires -n shoplist to target the correct namespace. The Service DNS name "postgres" works within the shoplist namespace. Cross-namespace DNS requires a fully qualified name.
COMMON MISTAKE
Forgetting -n shoplist on kubectl commands. Without it, kubectl targets the default namespace (empty). Nothing appears. Every kubectl command in this lecture must include -n shoplist.

Part 2 — Step-by-Step Instructions

Step 1

Create the k3d cluster

k3d cluster create shoplist --port "30080:30080@loadbalancer"

The --port flag maps host port 30080 to the cluster's load balancer.

kubectl cluster-info
kubectl get nodes

Expected: One node in Ready state.

Step 2

Import Docker images into k3d

k3d image import shoplist-backend:local shoplist-frontend:local -c shoplist

k3d runs inside Docker and has its own image cache. Images must be imported explicitly because the manifests use imagePullPolicy: Never — without this, pods show ErrImageNeverPull.

Interview Questions

Q1: What is the difference between a Pod and a Deployment in Kubernetes?
A Pod is a single running instance of one or more containers. A Deployment declares a desired number of pod replicas and manages their lifecycle — automatically creating replacement pods when existing ones fail. You always use Deployments, never bare pods, because bare pods have no restart mechanism.
Q2: What is the difference between a ClusterIP and a NodePort Service?
ClusterIP (the default) makes a Service reachable only from inside the cluster. NodePort exposes the Service on a specific port (30000–32767) on the cluster node, making it reachable from the host machine. We use NodePort only for the frontend; backend and postgres use ClusterIP.
Q3: Why do we use a Secret instead of a ConfigMap for database credentials?
Secrets can be access-controlled independently via Kubernetes RBAC — you can restrict which users and service accounts can read them. ConfigMaps are readable by any pod and user in the namespace. Note: base64 encoding is not encryption. The protection comes from access control, not from the encoding.
Q4: What is the purpose of a PVC, and what happens to it when a pod is deleted?
A PVC is a request for persistent storage that exists independently of any pod. When a pod is deleted, the PVC is not deleted — the Deployment creates a replacement pod that mounts the same PVC and finds all data intact. To permanently delete the data, you must explicitly run kubectl delete pvc postgres-pvc -n shoplist.
Q5: What does imagePullPolicy: Never do and why is it required with k3d?
It tells Kubernetes to never attempt to pull the image from a registry — use only what is already available in the cluster's local image cache. With k3d, images must be imported into the cluster with k3d image import before deploying. Without this setting, Kubernetes tries to pull from Docker Hub, fails, and the pod shows ErrImageNeverPull.

Appendix — Windows: Using Minikube Instead of k3d

Who this is for: Windows students who cannot use k3d (e.g. WSL2 is disabled on a corporate laptop). All 10 manifest files and all kubectl commands are unchanged. Only cluster setup, image loading, and browser access differ.

Install Minikube

Open PowerShell as Administrator:

winget install minikube

Verify: minikube version

Start the Cluster

minikube start

Minikube auto-selects a driver (Docker, Hyper-V, or VirtualBox). First run downloads ~300MB — takes 1–3 minutes once.

Load Images (instead of k3d image import)

Option A — build directly inside Minikube's Docker (recommended):

# Point your shell at Minikube's Docker daemon:
minikube docker-env | Invoke-Expression

# Then build:
docker build -t shoplist-backend:local app/backend/
docker build -t shoplist-frontend:local app/frontend/

Option B — load pre-built images:

minikube image load shoplist-backend:local
minikube image load shoplist-frontend:local
imagePullPolicy: Never is already set in the manifests — no YAML changes needed.

Apply Manifests

Identical to the main guide — run all 10 kubectl apply commands exactly as written in Steps 3–11.

Access the Application

NodePort services are not directly reachable on localhost:30080 with minikube. Use:

# Opens browser automatically to the correct URL:
minikube service frontend -n shoplist

Or for full localhost access (run in a separate admin terminal and keep it open):

minikube tunnel
# Then: http://localhost:30080 works as in the main guide
